# kb-compress — Предложение по архивному расширению

## Мотивация

Текущий процесс сжатия предлагает два варианта для устаревших записей: оставить или удалить.
Это психологически трудный выбор. Разработчики склонны оставлять всё «на всякий случай», поэтому индекс не уменьшается. Сжатие откладывается на неопределённый срок, а индекс накапливает низкосигнальные записи, которые снижают точность поиска.

**Архивный уровень** предоставляет третий вариант: убрать из активного поиска без постоянного удаления. Это снижает барьер для действий со стале записями и превращает сжатие в рутинную операцию обслуживания, а не в высокорисковое решение.

---

## Основная идея

Архивированные записи удаляются из активного `index.yaml` и перемещаются в отдельную директорию. Они не участвуют в обычном поиске. Их можно явно искать по запросу. Полное тело записи сохраняется.

Записи активного индекса, перенесённые в архив, заменяются **сжатой строкой** — минимальной заглушкой, которая сигнализирует о существовании записи в архиве без расхода бюджета поиска.

---

## Расширение структуры директорий

```
.knowledge/
├── index.yaml              <- только активные записи; архивированные записи отсутствуют
├── archive/
│   ├── index.yaml          <- архивный индекс; только сжатые строки
│   ├── issues/
│   │   └── kb-005.md       <- полное тело записи сохраняется
│   ├── tasks/
│   │   └── JIRA-4821.md
│   └── ...
```

`archive/index.yaml` следует той же схеме, что и основной `index.yaml`, но содержит дополнительное поле в каждой записи:

```yaml
- id: kb-005
  archived: true
  archived_date: 2026-03-17
  component: [user-service]
  summary: "Состояние пагинации сбрасывалось при навигации назад; исправлено в ALFA-32867."
  related: [ALFA-32867]
  original_file: issues/kb-005.md
```

Поля `triggers` и `tags` опускаются из архивного индекса — они больше не нужны для активного поиска, и именно их отсутствие делает архивный индекс лёгким.

---

## Формат сжатой строки (archive/index.yaml)

Минимально жизнеспособная строка, достаточная для поиска по запросу и ответа на вопрос «была ли у нас когда-нибудь похожая проблема?»:

| Поле | Обязательное | Примечания |
|------|--------------|------------|
| `id` | да | Тот же ID, что у исходной записи |
| `archived` | да | Всегда `true` |
| `archived_date` | да | Дата архивирования, формат `YYYY-MM-DD` |
| `component` | да | Сохраняется из исходной записи; полезен для поиска в архиве по компоненту |
| `summary` | да | Одно предложение; единственное содержимое, читаемое при поиске в архиве |
| `related` | при наличии | Сохраняется; позволяет прослеживать цепочки `superseded-by` / `fixed-by` в архив |
| `original_file` | да | Исходный путь относительно `.knowledge/`; используется для восстановления записи |

## Изменения алгоритма kb-compress

### Модифицированный Шаг 4 — Обнаружение устаревших записей (расширенный)

Текущие критерии для пометки записи как устаревшей (без изменений):
- `date` старше 12 месяцев
- Ни одна другая запись не ссылается на этот ID в своём поле `related`
- `version` всё ещё равен `1`
- Категория — `tasks` или `issues` (не `decisions` и не `behavior`)

Для каждой помеченной записи предлагать **три** варианта вместо двух:

```
Запись: kb-005  (2024-11-03, v1, issues)
Summary: "Состояние пагинации сбрасывалось при навигации назад."

Варианты:
  [k] Оставить — запись остаётся в активном индексе без изменений
  [a] Архивировать — переместить в archive/, заменить сжатой строкой
  [d] Удалить — удалить навсегда
```

**Никогда не предлагать архивирование для записей `decisions/`.** ADR намеренно постоянны и не должны архивироваться, даже когда они старые.

---

### Новый Шаг 4b — Поступенчатое архивирование для записей с `superseded-by`

Когда используются типизированные ссылки `related` (при активации предложения по улучшению), записи со ссылкой `rel: superseded-by` на более новую запись автоматически помечаются для архивирования независимо от возраста, поскольку более новая запись явно заменяет их.

Для каждой такой пары предлагать:

```
Запись: kb-003  (заменена JIRA-4821, v3)
Summary: "Исходное исправление пагинации; заменено обновлённой записью."

Варианты:
  [a] Архивировать kb-003 — рекомендуется, ссылка superseded-by присутствует
  [k] Оставить обе — если старая запись содержит отличающийся исторический контекст
  [d] Удалить kb-003
```

---

### Новый Шаг 7b — Запись архивных изменений

После всех подтверждений по каждому пункту, для каждой записи, подтверждённой к архивированию:

1. Переместить файл записи из текущей директории категории в `archive/<category>/`.
2. Удалить строку записи из активного `index.yaml`.
3. Добавить сжатую строку в `archive/index.yaml` (создать файл если отсутствует).
4. Если запись имела ссылки `related` из других активных записей, указывающие на неё:
   не удалять эти ссылки — они теперь корректно указывают в архив и позволяют прослеживать историческую цепочку.

## Поведение поиска для архивированных записей

### Обычный lookup (kb-lookup)

Архивированные записи **не ищутся** на Шагах 2–4 стандартного алгоритма lookup. `archive/index.yaml` никогда не открывается при обычном запросе.

Если lookup не возвращает результатов, резервное сообщение расширяется:

> "Записей по этой теме не найдено в базе знаний.
> Выполните `kb-expert: search archive — <тема>` для поиска в архивных записях.
> Хотите создать новую запись?"

### Явный поиск в архиве

Запускается при:
- `kb-expert: search archive — pagination reset`
- `kb-expert: find in archive — JIRA-4821`
- `kb-expert: что в архиве про nginx timeout?`

`kb-expert` загружает `kb-lookup.skill.md` и запускает тот же алгоритм против
`archive/index.yaml`, используя `summary` и `component` как цели поиска
(triggers и tags в архивном индексе отсутствуют).

Результаты возвращаются с чёткой пометкой:

> "Найдено в архиве (не в активной KB):
> kb-005 — Состояние пагинации сбрасывалось при навигации назад. [архивировано 2026-03-17]
> Выполните `kb-expert: restore kb-005` для возврата в активный индекс."

### Восстановление

`kb-expert: restore kb-005`

1. Переместить файл записи из `archive/issues/` обратно в `issues/`.
2. Восстановить полную строку активного `index.yaml` из frontmatter файла записи.
3. Удалить сжатую строку из `archive/index.yaml`.
4. Представить восстановленную строку на подтверждение перед записью.

## Ожидаемый коэффициент сжатия

На двухлетнем проекте с активным использованием KB примерно 30–40% записей `tasks` и `issues` становятся кандидатами на устаревание через 12 месяцев. При доступности архивирования:

- Активный `index.yaml` уменьшается на эту долю, снижая стоимость сканирования при запросе и частоту ложных срабатываний.
- `archive/index.yaml` содержит только `summary` + `component` на запись — примерно 3–4 строки против 8–12 строк в полной активной строке.
- Общие накладные расходы по хранению минимальны; полные тела записей сохраняются.

---

## Ограничения и нецели

- **Никогда не архивировать записи `decisions/`.** ADR — постоянные записи о намерениях.
- **Никогда не архивировать автоматически без подтверждения.** Всё архивирование — поштучное, подтверждённое пользователем, как и удаление.
- **Никогда не нарушать цепочки `related`.** Если активная запись A имеет `related: [kb-005]` и kb-005 архивирована, ссылка сохраняется. Ссылка теперь указывает в архив; следование по ней при обходе цепочки related читает из `archive/index.yaml`.
- Архив — не корзина. Записи, архивированные по ошибке, следует восстановить, а не оставлять накапливаться. Цель — небольшой, высокосигнальный активный индекс, а не постоянно растущий архив.
