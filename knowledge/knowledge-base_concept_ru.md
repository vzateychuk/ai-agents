# Концепция базы знаний

## Назначение

Этот документ описывает концепцию базы знаний на уровне проекта.

---

## Проблема

На протяжении жизни проекта накапливаются знания, которые нельзя извлечь из чтения исходного кода:

- Особенности деплоя, обнаруженные во время инцидентов
- Граничные случаи переменных окружения
- Ограничения конфигурации в реальных условиях
- Причины прошлых ошибок и способы их устранения
- Неочевидная бизнес-логика и поведение системы
- Принятые решения и их обоснование
- Завершённые задачи и конкретные файлы, которые были изменены

Эти знания сейчас хранятся в головах людей, в Slack-тредах, в комментариях к тикетам — или просто теряются. Загружать всё это в контекст AI на каждой сессии расточительно и зачастую невозможно.

Даже когда пользователь помнит, что похожая проблема уже решалась раньше, он не может эффективно передать этот контекст AI-ассистенту. AI начинает каждую сессию без памяти о прошлых решениях, прошлых ошибках и накопленных операционных знаниях. Он будет повторять те же ошибки, задавать те же уточняющие вопросы и пропускать неочевидные ограничения — если только эти знания не будут явно предоставлены.

---

## Концепция: RAG для накопленного опыта проекта

База знаний реализует лёгкий паттерн **Retrieval-Augmented Generation** (RAG)
для проектно-специфичного опыта.

Основная идея: когда пользователь задаёт вопрос, AI-ассистент не отвечает только на основе кода и общих знаний. Вместо этого он сначала извлекает релевантные записи — решения прошлых задач, причины известных ошибок, особенности конфигурации, архитектурные решения — и **инжектирует их как контекст** перед формулировкой ответа. Вопрос пользователя дополняется накопленным опытом проекта.

```
Вопрос пользователя
        ↓
kb-expert ищет в index.yaml
        ↓
Извлекает топ-N релевантных записей (макс. 3)
        ↓
Primary agent добавляет записи как контекст
        ↓
Ответ, обогащённый прошлым опытом проекта
```

### Слой выгрузки контекста

База знаний — это **слой выгрузки контекста** AI-агента. Контекстное окно AI-агента ограничено и не может хранить всю историю проекта на каждой сессии. KB решает эту проблему структурно: знания живут в файлах и инжектируются по требованию, в минимальном объёме, только когда релевантны. Именно поэтому правило `auto-consult` обязательно — стоимость пустого lookup минимальна (одно чтение `index.yaml`); стоимость пропущенного контекста — деградация ответа.

Когда команда понимает, что KB — это инфраструктура памяти агента, а не ещё одна wiki, мотивация к её наполнению меняется.

### Обзор архитектуры

![Обзор архитектуры](kb-architecture-ru.svg)
*Рисунок 1. Архитектура Knowledge Base*

**Ключевые свойства архитектуры:**

- `kb-expert` — суб-агент с единственной ответственностью: только чтение и запись в `.knowledge/`. Он не отвечает на вопросы, опираясь на вывод из кода.
- `index.yaml` — единственный файл, открываемый при поиске. Файлы записей открываются только для финальных 1–3 кандидатов, никогда при сканировании индекса.
- Каждая операция загружает ровно один skill-файл. Skills никогда не загружаются одновременно.
- Все операции записи требуют явного подтверждения пользователя перед изменением любого файла.

Это не система документации и не поисковый движок. База знаний не заменяет чтение кода — она фиксирует то, что код не говорит: почему было принято решение, как был устранён конкретный production-инцидент, какой граничный случай конфигурации приводил к тихому отказу. Именно эти факты делают разницу между общим ответом и правильным ответом для данного проекта.

В отличие от RAG с векторными базами данных, эта реализация не требует эмбеддингов, внешних сервисов и GPU. Индекс — YAML-файл. Поиск — сопоставление ключевых слов в полях `triggers` и `tags`. Это достаточно для баз знаний малого и среднего масштаба (до ~500 записей) и работает в любой среде, включая VDI.

Реализация совместима с любым AI-ассистентом, имеющим доступ к файловой системе.
Индекс намеренно хранится в одном файле — разбивка на подиндексы не даёт преимуществ, поскольку поиск ведётся по триггерам и тегам. Правильная стратегия масштабирования — сжатие индекса через `skills/kb/kb-compress.skill.md`, которое удаляет низкосигнальные записи, сохраняя качество поиска.

## Структура директорий

```
.knowledge/
├── index.yaml              <- компактный индекс, единственный файл при lookup
├── tags.md                 <- словарь одобренных тегов
├── deployment/           <- особенности деплоя: Docker, Helm, CI/CD
│   ├── kb-001.md
│   └── kb-002.md
├── config/               <- переменные окружения, конфигурация, секреты
│   ├── kb-003.md
│   └── kb-004.md
├── issues/                 <- известные ошибки, причины, обходные решения
│   ├── kb-005.md             <- kb-NNN, если нет трекерного тикета
│   └── JIRA-4102.md          <- ID тикета используется как имя файла
├── tasks/                <- завершённые задачи: что изменилось и почему
│   ├── kb-006.md
│   └── kb-007.md
├── behavior/             <- неочевидная бизнес-логика, поведение системы, граничные случаи
│   └── kb-008.md
├── decisions/            <- записи архитектурных решений (ADR)
│   ├── kb-042-auth-provider.md
│   └── kb-043-pagination-model.md
├── archive/              <- устаревшие записи, исключённые из активного поиска
│   ├── index.yaml
│   └── issues/
│       └── kb-005.md
└── misc/                 <- разное (bootstrap и вспомогательные записи)
    └── kb-000-knowledge-base.md
```

Категории можно расширять по мере роста проекта. Новые категории добавляются как новые директории верхнего уровня и документируются в `index.yaml`.

## Инициализация

Для нового проекта без `.knowledge/` первый шаг — инициализация базы знаний.

- Инициализация выполняется через суб-агент `kb-expert`:

  ```
  kb-expert: init kb
  ```

- Если `.knowledge/` уже существует:
  - Никакие файлы и директории не изменяются.
  - Агент сообщает текущий статус KB (какие основные файлы и категориальные директории присутствуют).

- Если `.knowledge/` не существует:
  - Создаётся `.knowledge/` и стандартные категориальные директории.
  - Создаётся `.knowledge/index.yaml` с:
    - `schema_version: 1`
    - `entries: []`
  - Создаётся `.knowledge/tags.md` с минимальным словарём тегов.
  - Создаётся `.knowledge/README.md` с кратким описанием и примерами команд `kb-expert`.
  - Создаётся bootstrap-запись `.knowledge/misc/kb-000-knowledge-base.md`, документирующая саму KB и способы её использования.
  - В `entries` файла `index.yaml` добавляется строка для `kb-000-knowledge-base`.
  - Если в корне репозитория присутствует `.gitignore` — в него добавляется правило для исключения `.knowledge/` из контроля версий; если `.gitignore` отсутствует, инициализация не создаёт его автоматически.

Если в корне проекта при инициализации существует `repo_map.md`:
- Из него может быть извлечён небольшой набор имён модулей/компонентов для наполнения `.knowledge/tags.md`:
  - приведены к нижнему регистру
  - пробелы и слэши заменены дефисами
  - добавлены только если отсутствуют и соответствуют правилам `kb-tags`
- Инициализация **не** создаёт дополнительных KB-записей напрямую из `repo_map.md`; он используется только для обогащения словаря тегов и bootstrap-записи.

## Соглашение об именовании ID

У каждой записи ровно один ID. Он используется в `index.yaml` и во frontmatter файла, из него выводится имя файла. Вторичных идентификаторов нет.

**Если существует тикет трекера** (`JIRA-1234`, `ALFA-32867`, `OCRV-654987` и т.д.):
используется ID тикета как есть. Это позволяет искать запись напрямую по номеру тикета без открытия файлов.

```
tasks/JIRA-4821.md       id: JIRA-4821
issues/ALFA-32867.md     id: ALFA-32867
```

**Если тикета трекера нет**: используется `kb-NNN` — единая глобальная последовательность независимо от директории. Категория определяется по директории — она не хранится ни в ID, ни во frontmatter записи.

```
issues/kb-003.md         id: kb-003
deployment/kb-001.md     id: kb-001
decisions/kb-042.md      id: kb-042
```

Для определения следующего номера: прочитать `index.yaml`, найти наибольшее значение `kb-NNN` среди всех записей, увеличить на 1, дополнить нулями до 3 цифр.

При добавлении новой категории создаётся соответствующая директория.
Таблица префиксов не нужна — имя директории несёт категорию.

### Именование файлов

Имена файлов записей выводятся из `id` и всегда имеют вид `kb-<sanitised-id>.md`.

- Начать с `id`.
- Каждый символ, не входящий в `[A-Za-z0-9_-]`, заменить на `-`.
- При необходимости схлопнуть последовательные `-` и убрать ведущие/завершающие.
- Если результат пустой — использовать запасной вариант, например `entry` или `entry-<number>`.

Примеры:

- `id: JIRA-4821` → файл `kb-JIRA-4821.md`
- `id: #1387` → файл `kb-1387.md`
- `id: ALFA#1387` → файл `kb-ALFA-1387.md`

## Формат index.yaml

`index.yaml` — единственный файл, который AI читает при выполнении lookup.
Он должен оставаться компактным. `summary` намеренно отсутствует в индексе; он хранится в файле записи и читается только когда нужно разрешить неоднозначность между кандидатами.

```yaml
schema_version: 1

entries:
  - id: JIRA-4821
    component: [user-service]
    related: []
    triggers:
      - email search missing
      - нет поиска по email
      - cannot find user by email
    tags:
      - 400
      - 401
      - 403
      - 500
    date: 2026-03-16

  - id: ALFA-32867
    component: [user-service]
    related: [JIRA-4821, JIRA-4102]
    triggers:
      - list resets on back
      - страница сбрасывается
      - pagination lost on back button
    tags:
      - reset
      - back-nav
      - pagination
      - v2api
    date: 2026-03-01

  - id: kb-001
    component: [infrastructure]
    related: []
    triggers:
      - docker build fails on arm
      - build error arm64
    tags:
      - crash
      - docker
      - arm64
      - multiarch
    date: 2026-02-10
    file: issues/kb-001.md
```

Правила полей:
- `id` — ID тикета трекера при наличии, иначе `kb-NNN`. Единственный идентификатор.
- `component` — список сервисов, модулей или инфраструктурных единиц, к которым относится запись. Допустимо несколько значений. Улучшает точность поиска при запросах по компоненту.
- `related` — список ID записей, причинно или тематически связанных с текущей. Пустой список, если нет.
- `triggers` — 2–6 фраз на естественном языке, как пользователь описал бы проблему (по правилу kb-tags). Включать не-английские фразы, если команда мультиязычна. Основная цель поиска.
- `tags` — 4–8 ключевых слов, выбранных **исключительно из `tags.md`** (по правилу kb-tags). Полный список тегов хранится в файле записи; индекс содержит только наиболее различимое подмножество.
- `date` — дата последнего обновления записи в формате `YYYY-MM-DD`. Совпадает с `date` во frontmatter. Позволяет быстро находить недавно обновлённые записи без открытия файлов. Новые и обновлённые записи должны включать `date`; старые индексы можно дополнять постепенно.
- `active` — опциональное булево поле. При значении `false` сигнализирует, что запись больше не актуальна (компонент переписан, решение заменено, конфиг изменился). По умолчанию отсутствует — считается `true`. Используется как фильтр поздней стадии при lookup, если кандидатов больше порога. См. раздел «Управление устареванием».
- `file` — опционально; путь к файлу записи (по умолчанию `kb-<sanitised-id>.md` в соответствующей директории категории). Если опущено, инструменты выводят путь из `id` по правилам именования файлов.

Таблицу в Markdown-формате можно сгенерировать из `index.yaml` по запросу:
`kb-expert: show index`.

### Словарь тегов

Теги хранятся в `.knowledge/tags.md` — плоском списке одобренных ключевых слов в нижнем регистре. Шаблон содержит минимальный начальный набор; добавляйте проектно-специфичные теги при создании записей. Количество, измерения и формат: см. правило `kb-tags`. Детальный алгоритм генерации тегов определён в `skills/kb/kb-write.skill.md` и является каноническим источником для поведения при выборе тегов.

### Измерения тегов

Каждая запись должна покрывать как можно больше из четырёх измерений, где применимо:

| Измерение | Что описывает                      | Примеры                                    |
|-----------|------------------------------------|--------------------------------------------|
| `symptom` | Наблюдаемое поведение, ошибка      | `crash`, `403`, `reset`, `timeout`, `null` |
| `module`  | Компонент, сервис или класс        | `role-guard`, `pagination`, `user-list`    |
| `tech`    | Технология или инфраструктура      | `auth0`, `helm`, `docker`, `arm64`, `jwt`  |
| `feature` | Функциональная область, сценарий   | `user-search`, `login`, `billing`          |

**Плохие теги** (только одно измерение — технические):
```
tags: [user, pagination, v2api]
```

**Хорошие теги** (несколько измерений, все из tags.md):
```
tags: [reset, back-nav, pagination, v2api, user-search]
#      ^сим   ^сим+мод  ^модуль    ^тех   ^фича
```

### Поле component

`component` — отдельное явное поле, а не тег — перечисляет сервисы, модули или инфраструктурные единицы, к которым относится запись. Одна запись может охватывать несколько компонентов; все должны быть указаны. Улучшает поиск при вопросах в разрезе компонента.

```yaml
component: [user-service]                     # один компонент
component: [user-service, users-controller]   # запись охватывает два сервиса
component: [infra]                            # нет единственного владельца
```

`component` обязателен в `index.yaml`. Во frontmatter записи — опционален, но рекомендован.

## Формат отдельной записи

Каждый файл знаний следует этой структуре:

```markdown
---
id: JIRA-4821
version: 1
summary: "Добавлено поле поиска по email в управление пользователями через UsersService."
component: [user-service]
tags: [missing-feature, user-list, user-service, manage-user, user-search]
triggers: ["email search missing", "нет поиска по email", "cannot find user by email"]
date: 2026-01-15
---

# Задача: Поиск пользователей по email

## Проблема
Поиск пользователей был возможен только по имени пользователя. Поиск по email отсутствовал.

## Решение
- Добавлено поле `email` в модель `SearchUserCriteria`
  (src/app/_models/search-user-criteria.ts)
- Расширен метод `searchUsers()` в `UsersService`
  (src/app/services/cis-users-service.ts:87)
- Добавлен ввод email в `ManageUserComponent`
  (src/app/components/user/manage-user/manage-user.component.ts:134)

## Примечания
Backend v2 API поддерживает email как query-параметр из коробки.
Изменения на бэкенде не потребовались.
```

**Справочник полей:**

| Поле          | Обязательное | Описание                                                                                              |
|---------------|--------------|-------------------------------------------------------------------------------------------------------|
| `id`          | да           | ID тикета трекера при наличии, иначе `kb-NNN`. Используется как имя файла.                           |
| `version`     | да           | Целое число, начиная с `1`. Увеличивается на 1 при каждом обновлении.                                |
| `summary`     | да           | Одно предложение, описывающее, о чём запись. Первое поле при RAG-инъекции.                           |
| `component`   | рекомендован | Список сервисов или модулей. Обязателен в index.yaml; опционален во frontmatter, но рекомендован.    |
| `tags`        | да           | Типизированные ключевые слова по измерениям symptom / module / tech / feature                        |
| `triggers`    | да           | Фразы на естественном языке — как пользователь описал бы проблему. Основная цель поиска.            |
| `date`        | да           | Дата последнего обновления в формате `YYYY-MM-DD`                                                    |
| `related`     | опционально  | Список ID причинно или тематически связанных записей. Опускается если нет.                           |
| `active`      | опционально  | Явно устанавливается в `false` (через kb-write, режим update), когда запись устарела. Отсутствие = активна. |

Поиск выполняется в порядке: `TRIGGERS` → `TAGS` → `SUMMARY` (только для кандидатов) → цепочка `RELATED` (только по явному запросу).
Поиск по `id` — точное или суффиксное совпадение (например, "1234" находит "JIRA-1234").

Разделы по категориям:

| Категория  | Разделы                                              |
|------------|------------------------------------------------------|
| tasks      | Проблема, Решение (со ссылками file:line), Примечания|
| issues     | Симптом, Корневая причина, Исправление, Файлы        |
| config     | Переменная/ключ, Назначение, Допустимые значения, Ловушки |
| deployment | Контекст, Проблема, Решение, Команды                 |
| behavior   | Наблюдение, Объяснение, Последствия                  |
| decisions  | Контекст, Решение, Последствия                       |

**Пишите записи как самодостаточные единицы.** Каждая запись должна быть понятна при чтении в изоляции AI-агентом без иного контекста о проекте. Не пишите «см. kb-001 для деталей» без копирования существенной детали в текущую запись.

## Записи архитектурных решений (ADR)

Архитектурные и дизайнерские решения, которые фиксируют контекст, невидимый в задачах или ошибках: почему была выбрана технология, какие альтернативы были отклонены, каких последствий ожидать.

Хранятся в `.knowledge/decisions/`. ID по соглашению `kb-NNN`. Опционально добавляйте slug для читаемости: `kb-042-auth-provider.md`.

ADR-записи никогда не удаляются при сжатии и никогда не архивируются — они являются постоянными записями о намерениях.

## Как AI использует базу знаний

### Процесс поиска (lookup)

Вопрос пользователя приходит на языке **симптомов и наблюдений**.
`index.yaml` — единственный файл при сканировании. Файлы записей открываются только для финальных кандидатов, никогда при сканировании.

**Правило раннего выхода:** после каждого шага, если остался ровно 1 кандидат → перейти сразу к Результату.

```
Пользователь: "список пользователей сбрасывается когда нажимаю кнопку назад"

─── Шаг 1: декомпозиция вопроса ─────────────────────────────────────────────
  Извлечь ключевые термины на всех языках:
    "reset", "сбрасывается", "user list", "back button", "back navigation"
  Вывести связанные технические термины: "pagination" (из "list + resets")

─── Шаг 2: сканирование TRIGGERS в index.yaml  [основной] ───────────────────
  Сопоставить извлечённые термины с полем triggers каждой записи.
  Также сопоставить с полем component, если вопрос называет сервис.
  Результат: kb-005, kb-007, JIRA-1234, JIRA-5501  (несколько кандидатов)
  → правило раннего выхода: >1 кандидат, перейти к Шагу 3

─── Шаг 3: сканирование TAGS в index.yaml  [сужение] ────────────────────────
  Запускается если Шаг 2 вернул 0 или более 1 кандидата.
  Результат: kb-005, JIRA-1234  (всё ещё несколько)
  → правило раннего выхода: >1 кандидат, перейти к Шагу 4
  → 0 кандидатов после Шага 3: перейти к резервному ответу

  Если кандидатов > 3 и некоторые записи имеют active: false:
  → сначала исключить записи с active: false
  → если все оставшиеся кандидаты active: false — не применять фильтр, предупредить пользователя

─── Шаг 4: чтение SUMMARY из файлов кандидатов  [развязка] ──────────────────
  Запускается если Шаг 3 оставил 2 или более кандидатов.
  Читать SUMMARY только из kb-005.md и JIRA-1234.md.
  "page size resets on back navigation" → kb-005 — лучшее совпадение
  → правило раннего выхода: 1 кандидат, перейти к Результату

─── Шаг 5: цепочка RELATED  [только по явному запросу] ──────────────────────
  "что вызвало X?": читать поле related из найденной записи, грепать index.yaml
  Рекурсивный обход до глубины 3.

─── Резервный ответ  [не найдено после всех шагов] ──────────────────────────
  "Записей по этой теме не найдено в базе знаний.
   Выполните `kb-expert: search archive — <тема>` для поиска в архиве.
   Хотите создать запись после решения проблемы?"
  НЕ изобретать знания и не угадывать из кода.

─── Результат ────────────────────────────────────────────────────────────────
  Прочитать файл(ы) найденных записей и ответить на основе верифицированных знаний.
```

### Модель вызова

Все операции записи требуют явного вызова пользователем.
Операции чтения следуют правилу ниже.

**Правило auto-consult:**

Primary agent ОБЯЗАН вызывать `kb-expert` перед ответом на любой нетривиальный вопрос. Нетривиальный — любой вопрос, который не является чисто общим знанием о языке или фреймворке (например, «что такое Java interface»).

Стоимость пустого lookup минимальна — одно чтение `index.yaml`. Стоимость пропущенного контекста — деградация ответа. При сомнении — всегда консультировать KB.

**Видимость:** auto-consult всегда виден пользователю:
> "Проверяю базу знаний на наличие релевантного контекста..."

Если `kb-expert` находит подходящие записи, primary agent включает их в ответ и цитирует ID:
> "Согласно записи ALFA-32867: ..."

Если `kb-expert` ничего не находит, primary agent явно сообщает об этом:
> "В базе знаний по этой теме ничего не найдено."
и продолжает с лучшим ответом из кода и контекста.

Пустой результат всегда показывается пользователю — это сигнал о том, что после решения проблемы стоит создать новую запись.

### Правило RAG-инъекции

Когда `kb-expert` возвращает записи, primary agent ОБЯЗАН использовать их как контекст, а не просто цитировать:

1. Прочитать каждую возвращённую запись полностью (frontmatter + тело).
2. Добавить записи к рассуждению как «контекст прошлого опыта» перед формулировкой ответа — обращаться с ними как будто пользователь только что объяснил предысторию.
3. Явно ссылаться на ID в ответе:
   > "На основании записи JIRA-4821: поиск по email был добавлен в CisUsersService, поэтому тот же паттерн применим здесь."
4. Если записи противоречат друг другу — не выбирать автоматически. Представить обе пользователю и явно спросить:
   > "Записи [ID-A] (v2) и [ID-B] (v1) противоречат друг другу по этой теме. Какая отражает текущее состояние? Я буду использовать ту, которую вы укажете."
5. **Максимум 3 записи на запрос.** При большем количестве использовать топ-3 по релевантности.
6. Если для ответа достаточно только `summary`, читать только его — не загружать полное тело. Это минимизирует расход контекстного окна.

База знаний управляется через явные триггеры:

**Триггер 1 — прямой запрос поиска:**
> "kb-expert: find details on JIRA-4821"
> "kb-expert: что мы знаем про проблему со сбросом пагинации?"

**Триггер 2 — прямой запрос записи:**
> "kb-expert: create entry for JIRA-5501, we fixed nginx timeout in prod"

**Триггер 3 — сигнал о завершении задачи:**

Когда пользователь сигнализирует о завершении работы над задачей:
> "done", "task complete", "JIRA-5501 closed", "закончил с задачей"

`kb-expert` не записывает немедленно. Вместо этого:

1. Реконструирует краткое описание сделанного из контекста сессии.
2. Представляет описание вместе с черновиком записи и запрашивает подтверждение.
3. При подтверждении — переходит к черновику через `skills/kb/kb-write.skill.md`.
4. При исправлении — учитывает исправления.
5. При отказе — принимает к сведению и не делает ничего.

**Правило подтверждения (все операции записи):** `kb-expert` всегда представляет полный черновик записи и строку обновления `index.yaml` перед записью чего-либо. Ни один файл не создаётся и не изменяется без явного подтверждения пользователя.

## Навыки создания записей

Операции с базой знаний реализованы в skill-файлах:
`skills/kb/kb-write.skill.md` — создание и обновление записей. `skills/kb/kb-lookup.skill.md` — поиск. `skills/kb/kb-compress.skill.md` — аудит и сжатие индекса. Один файл на операцию.

### Расположение skill-файлов

```
~/.agents/skills/
├── index.md               <- канонический entrypoint для обнаружения навыков
└── kb/
    ├── kb-write.skill.md      <- универсальный навык создания записей (все категории)
    ├── kb-lookup.skill.md     <- алгоритм поиска
    └── kb-compress.skill.md   <- сжатие и аудит индекса
```

## Управление устареванием

### Пометка записи как неактивной

Когда ситуация изменяется (компонент переписан, решение заменено, конфиг изменился):

1. Обновить запись через `kb-expert: update [ID]`, описав что изменилось.
2. `kb-write` устанавливает `active: false` во frontmatter записи и в строке `index.yaml`.
3. Запись остаётся в активном индексе и извлекается при поиске, но депривилегируется если кандидатов больше порога.

`active: false` устанавливается пользователем явно — `kb-write` не предлагает это поле при создании записи и не устанавливает автоматически. Сигналы для установки: «это решение больше не применяется», «сервис переписан», «конфиг изменился», «гипотеза не подтвердилась».

Записи с `active: false` никогда не удаляются автоматически. Удаление или архивирование — отдельный шаг.

### Обновление записи

При изменении ситуации (ошибка исправлена, конфиг обновлён, поведение изменилось):

1. Отредактировать файл записи на месте с новым содержимым.
2. Увеличить `version` на 1 во frontmatter.
3. Обновить `date` на сегодняшнюю дату.
4. Обновить соответствующую строку в `index.yaml`: синхронизировать `triggers`, `tags`, `component`, `related`, `active` при изменении любого из них.

### Архивный уровень

`kb-compress` предлагает третий вариант помимо «оставить» и «удалить»: **архивировать**.

Архивированные записи перемещаются из активного `index.yaml` в `archive/`. Они не участвуют в обычном поиске. Полное тело записи сохраняется. Сжатая заглушка (id, summary, component, archived_date) записывается в `archive/index.yaml`.

Резервное сообщение при отсутствии активных записей:
> "Записей по этой теме не найдено в базе знаний.
>  Выполните `kb-expert: search archive — <тема>` для поиска в архиве."

Записи можно восстановить явно:
> `kb-expert: restore kb-005`

**Записи `decisions/` никогда не архивируются.** ADR — постоянные записи о намерениях.

Полная спецификация архивного уровня: `kb-compress_archive-proposal.md`.

## Сжатие индекса

Запустить `kb-compress` когда:
- Пользователь явно запрашивает: `kb-expert: compress index`
- Количество записей в `index.yaml` превышает ~500
- Качество поиска заметно снижается (слишком много ложных срабатываний)

`kb-compress` выполняет аудит в следующем порядке, запрашивая подтверждение по каждому пункту перед применением изменений:

1. Подсчёт записей и сообщение статуса.
2. Обнаружение дублирующихся триггеров — предложение слияния, дифференциации или удаления.
3. Обнаружение потери RAG-слотов — записей, которые совместно появляются в каждом топ-3 для одного запроса.
4. Обнаружение избыточных тегов внутри записей.
5. Обнаружение устаревших записей (`date` > 12 месяцев, `version: 1`, категория `tasks` или `issues`, нет ссылок).
6. Обнаружение раздутых триггеров (более 6 на запись).
7. Для каждой устаревшей записи: предложить оставить / архивировать / удалить (три варианта, а не два).
8. Применение подтверждённых изменений и отчёт о старых и новых размерах.

Записи `decisions/` никогда не удаляются и не архивируются `kb-compress`.

## Проверка целостности

`kb-verify` — отдельный навык для плановых проверок целостности. Не является частью обычного процесса lookup и не запускается автоматически.

Два режима:
- `kb-expert: verify kb` — только отчёт, ничего не изменяется
- `kb-expert: verify-fix kb` — отчёт + автоматические исправления где возможно

`kb-verify` проверяет пути к файлам в телах записей (шаблон `something.ext:NNN`), целостность цепочек `related` и дрейф пространства имён компонентов. В режиме `verify-fix` пытается разрешить устаревшие пути к файлам, ища файлы с тем же именем в текущем дереве проекта, предлагая замены для подтверждения.

`kb-verify` также выявляет кандидатов на пометку `active: false` — записи старше настраиваемого порога без обновлений. Список представляется пользователю для ревью; флаг никогда не устанавливается автоматически.
